üéØ Why HTTP Verbs Are Important in REST APIs

They allow you to build clean, meaningful routes:

HTTP Verb	Path	Meaning
GET	/users	fetch all users

This keeps your API predictable, scalable, and standardized.

1. What are HTTP verbs, and why are they used?

They are methods (GET, POST, PUT‚Ä¶) that tell the server what operation the client wants to perform on a resource.
***********************************************************************

. What is idempotent in HTTP?

An operation giving the same result no matter how many times it‚Äôs repeated.

Idempotent methods: GET, PUT, DELETE, HEAD, OPTIONS
Not idempotent: POST
**********************************************************************
7. Safe vs unsafe HTTP methods?

Safe: GET, HEAD ‚Üí don‚Äôt modify resources

Unsafe: POST, PUT, DELETE, PATCH ‚Üí change server state

**********************************************************
8. What are HTTP status codes?

Codes representing the result of a request, grouped into:

1xx ‚Üí Info

2xx ‚Üí Success

3xx ‚Üí Redirect

4xx ‚Üí Client error

5xx ‚Üí Server error

*********************************************. When to use 200 vs 201 vs 204?

200 OK: Successful GET

201 Created: New resource created

204 No Content: Success but no response body
****************************************************
10. What is OPTIONS used for?

To check allowed HTTP methods for a route; used in CORS preflight.

11. What is HEAD used for?

Same as GET but returns headers only ‚Äî no body. Used for cache validation.

12. What is a REST API?

An API following REST constraints: stateless, resource-based, uniform interface.

13. REST vs RESTful?

REST = theory/architecture
RESTful = real implementation following REST constraints

14. What does ‚Äústateless‚Äù mean?

Server does not store session data. Each request carries all required info (token, auth).

15. Resource naming rules?

Use plural nouns:

‚úî /users

16. Why avoid verbs in URL paths?

HTTP verbs already indicate actions.
URL should represent nouns/resources, not actions.
